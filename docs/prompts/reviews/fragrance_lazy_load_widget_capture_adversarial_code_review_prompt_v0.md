## Implementation Summary To Verify

Commit `2c08ebb6` is intended to add bounded lazy-load trigger support to the
existing rendered+widget companion route:

- adds `lazy_load_scroll_passes` and `lazy_load_scroll_step_px` to the browser page observation protocol and public fetch function;
- rejects negative lazy-load scroll controls;
- keeps the default at zero passes, preserving the original no-scroll behavior unless the operator opts in;
- extracts visible text and DOM observation before any lazy-load scroll, then runs bounded scroll passes and only then reads observed widget responses;
- records `dom_observation_stage: pre_lazy_load_scroll`, requested scroll passes, step size, and executed pass count in observation metadata;
- caps requested scroll passes at `_MAX_SCROLL_PASSES` and emits a warning note when capped;
- uses step scrolling when `lazy_load_scroll_step_px` is positive and bottom scrolling otherwise;
- threads the lazy-load controls through `capture_fragrance_rendered_widget_companion`, `write_fragrance_rendered_widget_companion`, and the runner CLI;
- adds route-health markers for `lazy_load_scroll_passes_requested:N` and `lazy_load_scroll_passes_executed:N` when the requested pass count is positive;
- does not add DOM review-row parsing, new live capture semantics, new fallback URL discovery, or source-wide review coverage claims;
- adds synthetic tests for adapter threading, negative-value validation, bounded scroll behavior, cap behavior, companion threading, route-health markers, and runner threading.

The intended operator semantics are one command / one capture run that can render
the PDP and preserve widget responses fired by bounded lazy-load scrolls during
that same browser session. It is not one literal HTTP request. Fallback direct
widget URLs remain a bounded completion mechanism only when widget rows are still
missing or incomplete.

## Review Questions

Be adversarial about material implementation risks:

1. Does the adapter truly preserve no-scroll above-fold DOM/text facts before any lazy-load scroll can mutate the rendered page state?
2. Does lazy-load scrolling broaden only observed widget responses, without mutating `rendered_companion` facts or implying below-fold DOM row capture?
3. Are the metadata and route-health markers sufficient to compare a no-scroll baseline against an opt-in bounded-scroll capture?
4. Can requested pass caps be hidden from the companion receipt because the warning note remains in adapter warnings rather than route health?
5. Does `_run_bounded_lazy_load_scrolls` avoid unbounded or infinite scroll behavior on long pages, virtualized pages, sticky containers, or pages with expanding body height?
6. Is bottom-scroll-per-pass behavior too coarse for widgets that load only after stepwise viewport intersections, and is the `--lazy-load-scroll-step-px` interface clear enough?
7. If `position >= document.body.scrollHeight` before reaching a review widget, can the step-scroll path prematurely report fewer executed passes without a residual that downstream readers notice?
8. Does the response listener remain active long enough after each scroll to observe lazy widget calls, and is `_SCROLL_PASS_SETTLE_MS` defensible without operator control?
9. Does the patch preserve fallback semantics: bounded fallback requests happen only after the render/passive route still cannot complete row coverage?
10. Does the receipt avoid claiming DOM review-row extraction, widget completeness, source-wide review coverage, or one literal request?
11. Are tests strong enough for Playwright ordering, cap warnings, pass execution count, metadata propagation, route health, runner CLI parsing, and default zero-pass backward compatibility?
12. Does adding lazy-load arguments to the generic browser page observation adapter create backward-compatibility risk for other callers or fake success paths through permissive defaults?
13. Are warning and limitation notes from the lazy-load stage visible enough when a capture partially works but lazy widget observation silently fails?
14. Should a future patch add a post-scroll DOM observation as a separate origin, or is keeping the current row source limited to widget responses plus fallback the safer MGT boundary? Assess risk, not implementation.
15. Does the current design correctly avoid ECR, Cleaning, Judgment, sentiment, pain/pleasure, integrity, credibility, demand, buyer-proof, validation, readiness, and fixture-admission claims?
