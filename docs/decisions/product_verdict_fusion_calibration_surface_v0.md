## What this is, and is not

"Calibrating" the product-verdict fusion does **not** mean training a model. The fusion in
`scoring/product_fusion.py` is a deterministic rule system: six constants plus one
accumulation law. The only thing that makes a constant "right" is agreement with the owner's
own verdict on **real** mentions — and no labeled set of that exists yet. So this record names
the surface and the current defaults' consequences; choosing values is owner-gated and corpus-gated.

## The knob surface

These six fields in the `FusionConfig` dataclass are the entire tunable surface. The defaults
were chosen by mirroring `scoring/audience_fusion.py` — a principled prior, **not** a value
fit to owner-labeled data.

| Knob | Default | Controls | Type |
|---|---|---|---|
| `material_min` | 0.40 | a side (support/oppose) counts only if its squashed strength >= this — the abstain/decide line | risk dial (owner preference) |
| `gain` | 2.0 | tanh squash speed: `side = tanh(gain * raw)` — how fast accumulated evidence saturates | shape |
| `creator_authored_mult` | 1.0 | weight of an explicit / creator-authored stance | shape (ratio) |
| `implicit_mult` | 0.6 | weight of an inferred / implicit stance | shape (ratio) |
| `negation_mult` | 0.3 | discount applied when irony / negation is flagged | shape |
| `dependence_exponent` | 0.5 | within-video repeat discount `N ** -exponent` (0.5 == 1/sqrt(N)) | shape |

The verdict map itself (both sides material -> mixed; one -> positive/negative; neither ->
unknown) is structural, governed entirely by `material_min`. There is **no engagement /
resonance knob** — engagement is the deferred Judgment brain, out of calibration scope.

## The shared-surface catch (owner-pending)

`product_fusion` did not pick these numbers — it **copied** them from `audience_fusion`:
`gain` 2.0, the abstain floor 0.40, and the `1.0 / 0.6 / 0.3` multipliers are identical across
both Pass-2 fusions. So neither is calibrated; both are the same v0 prior.

**Owner decision:** are "authored beats implicit 1.0:0.6" and "abstain below 0.40" *one shared
policy* (calibrate once, keep the two modules in sync, possibly lift to a shared config) or two
independent policies (allowed to diverge, but divergence should be a recorded decision, not
copy-paste drift)? Not resolved here.

## What the current defaults actually do

are PROPERTIES OF THE CONSTANTS, true regardless of which data they run on; they are not
calibration data and do not pick any value):

Single-witness bar [idealized, pre-rounding] (min stance*conf that decides a lone authored verdict): 0.2118

|---|---|---|---|---|
| stance 0.2 | unk | unk | unk | unk |

| stance 0.7 | unk | unk | unk | unk |
| stance 1   | unk | unk | unk | pos |

Three findings fall out:

1. **It is a single-witness system at a low bar.** One authored mention with stance x confidence
   at about 0.2118 (idealized; the 4-dp rounding puts the operational bar a hair lower) already
   decides a confident verdict — no corroboration required (authored boundary
   grid: stance 0.3 x conf 1.0 -> positive). Whether one creator statement should be enough is
   the biggest owner call buried in `material_min` / `gain`.
2. **The irony flag alone does not abstain a strong mention.** At `negation_mult` 0.3 a
   flagged-ironic but max-stance/-confidence mention still scores 0.537 -> positive (irony grid:
   only the stance 1.0 x conf 1.0 corner clears). The discount works only because the *extractor*
   is also expected to lower `stance_vote` for sarcasm; the deterministic floor does not enforce
   it. To make the flag alone abstain a max mention, `negation_mult` would need to fall below the
   single-witness bar (~0.2118, pre-rounding): at max stance x confidence the contribution equals
   `negation_mult`, so it clears exactly when that bar is exceeded. (0.15 is merely the nearest
   sampled point below it in the harness sweep, not the boundary.)
3. **Saturation hides corroboration.** 1 strong mention -> 0.964; 2 -> 0.999; 3 -> 1.0. Past the
   first strong mention more evidence barely moves the score, so `uncalibrated_support_score` is
   near-binary, not an "endorsement intensity" signal. If Judgment wants intensity it needs
   `mention_count` / the raw pre-squash sum, not the squashed score. `gain` sets where saturation
   begins.

## Why the six numbers cannot just be nudged

- **No ground truth.** Calibration means matching owner labels; there are none. The current unit
  tests are synthetic property tests (authored > implicit, ironic < plain) — they assert
  inequalities, and a few pin the current floor's boundary outcomes as expected answers, so today
  the suite encodes *our* assumptions, not the owner's.
- **Six constants cannot be fit on a handful of reels** without overfitting. Honest v1 is
  "defaults + this sensitivity table + owner sign-off on the risk dial", not statistical
  optimization.
- **Entanglement with Pass-1.** A constant only means something relative to how the *real*
  extractor distributes `stance_vote` / `extractor_confidence`. Calibrating on synthetic mentions
  re-encodes our priors — the "synthetic data is worthless" trap. Real calibration must run on
  real extracted mentions.

## Provenance honesty

`ProductVerdictSet.provenance` records `fusion_config_version`. Because the new public `config`
parameter could otherwise let a non-default run be mislabelled by the version string alone,
provenance now ALSO records the actual `FusionConfig` values whenever `config` differs from
`DEFAULT_FUSION_CONFIG` (the default path is unchanged, preserving Phase A behaviour). Still owed
in Phase C: when the DEFAULT constants are recalibrated, bump `FUSION_CONFIG_VERSION` so the
default-path version string stays meaningful.
